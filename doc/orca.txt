*orca.txt*  Branch review for orca deliverables, in your own editor.

                                                        *orca* *orca.nvim*

orca.nvim opens a branch review — the merge-base diff of a deliverable
branch against trunk — inside your fully-configured Neovim: your LSP, your
colors, your muscle memory. It is the human half of review; an orca run's
independent reviewer already attacked the diff mechanically, and this is
the look-through before `git merge --no-ff`. Comments you write on the way
persist under `.orca/` and flow back into the run (|orca-notes|).

It requires an orca-managed repository: `.orca/` must exist at the repo
root. In a repo without one, |:OrcaReview| refuses with a pointer at
`/orca:init`. No required setup(), no dependencies.

==============================================================================
COMMANDS                                                        *orca-commands*

                                                                  *:OrcaReview*
:OrcaReview [{range}]
        Start (or restart — at most one session exists) a review session.
        {range} is `<base>...<head>`, always in the three-dot merge-base
        sense: only what the branch changed, never what trunk gained since
        the branch point. A bare `<base>` implies `...HEAD`; no argument
        defaults to `<trunk>...HEAD`. Trunk resolves in tiers: in a
        linked worktree, the symbolic HEAD of the common git dir (the
        bare repo's HEAD in orca's bare-with-worktrees layout); in a
        normal checkout, the branch `origin/HEAD` names (preferring its
        local twin), else the first of `main`/`master`/`trunk` that
        exists locally.

        The changed files populate the |quickfix| list (`M`/`A`/`D`/`R`
        status in each entry) and the first file's diff pair opens
        immediately. <CR> on a quickfix entry opens that file's pair:
        the working-tree file on the right — a real buffer, LSP attaches,
        editable — and the merge-base version on the left in a read-only
        scratch buffer. Native diff mode drives both, so |]c|, |[c|, |do|
        and |dp| work as everywhere else.

        Any existing review notes for the branch load with the session
        (|orca-notes|): comments from an earlier sitting reappear at
        their anchors, and resolutions orca wrote back render under them.

        Edge cases: added files diff against an empty left side; deleted
        files show the removed content on the left with an empty right
        (there is no working-tree file); renames are detected and load the
        old path at the merge-base; binary files open plainly, marked
        `(binary)` in the list. A dirty working tree is diffed as-is —
        what you review is what would merge.

        The pair follows navigation. Entering a changed file by any route
        — a fuzzy finder, |gd|, plain |:edit| — opens its diff pair around
        the window you landed in, quickfix selection synced. Wandering off
        to a buffer outside the review collapses the pair (left split
        closed, diff mode off); the session stays alive, and the quickfix
        window, |:OrcaReviewNext|, or entering any changed file resumes
        the walk.

                                                             *:OrcaReviewClose*
:OrcaReviewClose
        End the session: review notes saved, diff mode off, scratch
        buffers wiped, keymaps removed. The quickfix list is left in
        place — it is yours. Quitting Neovim cleans up the same way
        (scratch buffers are |nofile| + `bufhidden=wipe`; the only thing
        that persists is the notes file, and persisting is its job).

                                            *:OrcaReviewNext* *:OrcaReviewPrev*
:OrcaReviewNext
:OrcaReviewPrev
        Move to the next/previous changed file, tearing down the current
        pair and opening the new one.

                                                                 *:OrcaComment*
:[range]OrcaComment
        Create or edit the review comment anchored at the cursor line —
        or, with a visual selection or {range}, at that line range. On a
        line an existing comment covers, the comment opens for editing
        (which re-opens it: changed text means any previous resolution no
        longer answers it).

        Input happens in a small scratch split at the bottom — review
        comments are multi-line. `:w` (or `:wq`) commits; quitting
        without writing aborts; committing empty text deletes the
        comment. Each committed comment is written to the notes file
        immediately (|orca-notes|) — a crash loses nothing.

        Comments anchor to the working-tree (right) side of the pair
        only. The left buffer is a base-version scratch; a comment there
        ("this deletion was wrong") has no stable working-tree anchor and
        is politely refused, as are deleted and binary entries.

                                                           *:OrcaCommentDelete*
:OrcaCommentDelete
        Delete the comment covering the cursor line. (Committing empty
        text from |:OrcaComment| does the same.)

REVIEW NOTES                                                       *orca-notes*

Comments persist to a findings-adjacent JSON file that orca (the Claude
Code plugin's skills) consumes: >

        .orca/review-notes/<key>.json
<
`<key>` is the head branch with anything outside `[A-Za-z0-9._-]` replaced
by `-` (or the sanitized full range when head is not a branch — detached
HEAD, a sha). The repo root is the parent of the common git dir — the same
rule the orca skills use, so it works from inside any worktree of orca's
bare-repo layout and in a plain checkout alike. Everything is derived from
git state: quitting Neovim on Tuesday and reopening the worktree on
Wednesday finds the same file, with no launch-time handoff. The file is
created lazily on the first comment — a comment-less review leaves no
trace — and deleting the last comment removes it.

While the session lives, each comment is an extmark, so anchors ride
buffer edits (fixing nits during review is a feature). Every create, edit,
and delete rewrites the whole file from the extmarks' current positions —
always a complete snapshot, never an append. Alongside each line number
the file records the anchored line's text (`quoted`): numbers rot as soon
as fixes land, and the quoted text is how orca's fix agent finds the spot
after drift.

Comment text renders as virtual lines under the anchor, soft-wrapped to
the window's text width. Wrapping is display-only — recomputed at every
placement and on window resize, with the stored text never touched — and
your own line breaks are kept as paragraph breaks. A range comment shows
the `┃` gutter marker on every line it covers (Neovim 0.10+; older
versions mark the anchor line only).

The round trip: orca's addressing step converts each `open` comment into a
finding (severity High — if you bothered to write it, it matters), runs
the fix machinery, and writes `status` — `addressed`, or `answered` for
questions — plus a `resolution` note back into the same file, archiving a
copy with the run's artifacts. The next |:OrcaReview| renders resolutions
inline under their anchors. One file per branch is safe because the
workflow is sequential: orca runs, you comment, orca addresses, you
re-review — one writer at a time.

The file carries a `version` field (currently 1): plugin and skill ship
separately, and either side fails loud on a version it doesn't know —
orca.nvim then disables commenting rather than clobber a newer file. If
the file's recorded head is not an ancestor of the branch under review (a
recycled branch name), the load warns; delete the file to start fresh.

Highlights, all with sensible defaults you can override:
`OrcaCommentSign` (the `┃` sign column marker), `OrcaCommentText` (the
comment's virtual lines), `OrcaCommentResolution` (orca's write-back).

KEYMAPS                                                          *orca-keymaps*

Buffer-local only, set in the buffers the session owns (both diff sides and
the quickfix window) and removed when the session lets go of them. The
commands above are the public API; map your own globals if you want them.

Every default is a native key upgraded in place — orca never binds a key
that doesn't already mean what orca makes it do:

        ]q              :OrcaReviewNext — |]q|'s own meaning, direct route.
                        Takes a count: `3]q` moves three files, clamped at
                        the list edge.
        [q              :OrcaReviewPrev, same contract.
        <CR>            (quickfix window only) open this file's diff pair

If another list lands in the quickfix window mid-session (|:grep|, LSP
references), the keys fall back to stock behavior — <CR> jumps to the
entry, ]q/[q run |:cnext|/|:cprevious| — until orca's list is current
again.

|]c| and |[c| hunk motions are native diff mode — nothing to define.

Comment and close ship unbound: no native key means what they do, and
<leader> is yours. They stay available as |:OrcaComment| /
|:OrcaReviewClose|; one line binds them if you want keys: >

        vim.g.orca_mappings = { comment = '<leader>rc', close = '<leader>rq' }
<
A `comment` binding maps normal mode (cursor line) and visual mode (the
selected range).

                                                          *vim.g.orca_mappings*
`vim.g.orca_mappings` reshapes the maps, read when a session starts (order
relative to plugin load never matters). A table overrides per action — keys
`next`, `prev`, `comment`, `close`, `open` — with `false` dropping that one
map; the whole variable set to `false` drops them all: >

        vim.g.orca_mappings = { next = ']f', prev = '[f' }   -- the old keys
        vim.g.orca_mappings = false                          -- commands only
<
                                                                  *orca.setup()*
setup({opts}) is optional sugar over the same variable — `setup({mappings =
...})` assigns `vim.g.orca_mappings` and nothing else, so lazy.nvim's
`opts` works: >

        {
          "miguelbacalhau/orca.nvim",
          opts = { mappings = { comment = "<leader>v" } },
        }
<
Nothing requires calling it; the plugin stays lazy either way.

==============================================================================
INSTALLATION                                                     *orca-install*

Install with your plugin manager, like any other plugin. lazy.nvim: >

        { "miguelbacalhau/orca.nvim" }
<
No setup() call required, no dependencies — |orca-keymaps| covers the only
configuration. Any manager that installs from GitHub works the same way;
updates ride the manager's own update flow.

Without a plugin manager, clone into Neovim's native packpath
(|packages|): >

        git clone https://github.com/miguelbacalhau/orca.nvim \
            ~/.local/share/nvim/site/pack/orca/start/orca.nvim
<
(the parent directory is `stdpath('data')` — check with
`:echo stdpath('data')` if yours differs), then generate help tags with
`:helptags ALL`. Updating is `git pull` in the clone; uninstalling is
removing the directory. Caveat: native packages need 'packpath' intact —
configs that reset it (some plugin managers do) ignore the clone
silently, in which case the manager route above is the fix.

If you use orca (the Claude Code plugin), `/orca:doctor` checks the
install and prescribes whichever path fits.

`:checkhealth orca` verifies the setup: git ≥ 2.5 on PATH, inside a git
repository, `.orca/` at the repo root, trunk resolvable.
